# ABAP MESSAGE Statement in Function Modules
**Reference:** WO Approval System — `ZFM_APPR_CHK_REL` & Exit Pattern  
**Date:** February 2026

---

## 1. Overview

In ABAP, the `MESSAGE` statement inside a **Function Module** works differently from a normal program. A Function Module communicates errors/info back to the **caller** via:

1. `RAISING` — triggers an exception that the caller handles via `sy-subrc`
2. `sy-msg*` system fields — message details passed back automatically
3. Caller re-raises the message using `MESSAGE ID sy-msgid TYPE sy-msgty ...`

---

## 2. Syntax in Function Module

### Pattern A — MESSAGE … RAISING (Most Common)

```abap
FUNCTION zfm_appr_chk_rel.
*"  EXCEPTIONS
*"      INSERT_FAILED
*"      BLOCKED

  " --- Trigger exception with a message ---
  MESSAGE i000(db) WITH 'Release blocked' 'Use ZR_SVC_WO_APPROVAL'
    RAISING blocked.

  " --- Or for error ---
  MESSAGE e000(db) WITH 'Failed to create header for WO' iv_aufnr
    RAISING insert_failed.

ENDFUNCTION.
```

**How it works:**

| Element | Description |
|---|---|
| `i000(db)` | Message class `db`, number `000`, type `I` (Info) |
| `e000(db)` | Same class/number, type `E` (Error) |
| `WITH '...' iv_aufnr` | Up to 4 message variables (populated into sy-msgv1..v4) |
| `RAISING blocked` | Sets `sy-subrc = 2` (position of exception in EXCEPTIONS list) |

> **Key Rule:** `MESSAGE ... RAISING` does **not** display any dialog to the user.  
> It only populates `sy-msg*` fields and sets `sy-subrc`. The caller decides what to show.

---

### Pattern B — MESSAGE without RAISING (Display + Return)

```abap
" Use only in executable programs (reports), NOT in FMs called from BAdI/Exit.
" In FMs: prefer RAISING to let caller control the flow.
MESSAGE e398(00) WITH 'Approval Required for WO' lv_aufnr ''.
```

This **does** display a message immediately and is typically used when the FM *is itself* the main program entry point or a screen event handler.

## 3. How the Caller Handles It (Exit / BAdI)

From `Exit-WO.txt` — the caller pattern:

```abap
" Step 2: Check/Create ZTWOAPPRH header
CALL FUNCTION 'ZFM_APPR_CHK_REL'
  EXPORTING
    iv_aufnr       = caufvd_imp-aufnr
    iv_werks       = caufvd_imp-werks
    iv_gsber       = caufvd_imp-gsber
    iv_auart       = caufvd_imp-auart
  IMPORTING
    ev_allowed     = lv_ev_allowed
    ev_appr_status = lv_ev_appr_status
  EXCEPTIONS
    insert_failed  = 1       " sy-subrc = 1
    blocked        = 2       " sy-subrc = 2
    OTHERS         = 3.

IF sy-subrc <> 0.
  " Re-raise the message that the FM set via MESSAGE ... RAISING
  MESSAGE ID sy-msgid TYPE sy-msgty NUMBER sy-msgno
            WITH sy-msgv1 sy-msgv2 sy-msgv3 sy-msgv4.
  no_release = 'X'.
ENDIF.
```

**What happens step by step:**

```
FM: MESSAGE i000(db) WITH 'Blocked' 'Use Report' RAISING blocked.
    ↓
    sy-subrc = 2
    sy-msgid = 'DB'
    sy-msgty = 'I'
    sy-msgno = '000'
    sy-msgv1 = 'Blocked'
    sy-msgv2 = 'Use Report'
    ↓
Caller: IF sy-subrc <> 0.
    ↓
    MESSAGE ID 'DB' TYPE 'I' NUMBER '000'
              WITH 'Blocked' 'Use Report' ...
    ↓
    → User sees the Info/Error popup
```

## 4. Exception List Numbering

The position in the `EXCEPTIONS` list determines `sy-subrc`:

```abap
CALL FUNCTION 'ZFM_APPR_CHK_REL'
  ...
  EXCEPTIONS
    insert_failed  = 1    " MESSAGE ... RAISING insert_failed → sy-subrc = 1
    blocked        = 2    " MESSAGE ... RAISING blocked        → sy-subrc = 2
    OTHERS         = 3.   " Any other unhandled exception      → sy-subrc = 3
```

Always define `OTHERS` to catch unexpected exceptions safely.

---

## 5. Message Types Reference

| Type | Code | Behavior in FM (with RAISING) |
|---|---|---|
| Error | `E` | Sets sy-msgty = 'E'; caller re-raises as error (stops action) |
| Warning | `W` | Sets sy-msgty = 'W'; caller can re-raise as warning |
| Info | `I` | Sets sy-msgty = 'I'; caller shows info popup |
| Abort | `A` | Terminates program — avoid in FMs |
| Success | `S` | Status bar message — rarely used with RAISING |

> **Best practice for WO Approval:** Use type `E` for `insert_failed` and type `I` for `blocked`  
> (shows informational popup, not a hard error dump).

## 6. Complete FM Template (WO Approval Pattern)

```abap
FUNCTION zfm_appr_chk_rel.
*"----------------------------------------------------------------------
*"  IMPORTING
*"     REFERENCE(IV_AUFNR) TYPE  AUFNR
*"     REFERENCE(IV_WERKS) TYPE  WERKS_D
*"  EXPORTING
*"     REFERENCE(EV_ALLOWED) TYPE  CHAR1
*"  EXCEPTIONS
*"      INSERT_FAILED
*"      BLOCKED
*"----------------------------------------------------------------------

  DATA: ls_header TYPE ztwoapprh,
        lv_status TYPE char1.

  CLEAR ev_allowed.

  " --- Read header ---
  SELECT SINGLE appr_status FROM ztwoapprh
    INTO lv_status
    WHERE aufnr = iv_aufnr.

  IF sy-subrc <> 0.
    " First time: create header
    ls_header-aufnr       = iv_aufnr.
    ls_header-appr_status = '1'.
    ls_header-werks       = iv_werks.
    INSERT ztwoapprh FROM ls_header.

    IF sy-subrc <> 0.
      " Error message → caller gets sy-subrc = 1
      MESSAGE e000(db) WITH 'Failed to create approval header for WO'
                             iv_aufnr RAISING insert_failed.
    ENDIF.

    COMMIT WORK AND WAIT.
    lv_status = '1'.
  ENDIF.

  CASE lv_status.
    WHEN '2'.
      ev_allowed = 'X'.   " Allow release — no message needed

    WHEN '1'.
      " Info message → caller gets sy-subrc = 2
      MESSAGE i000(db) WITH 'Release blocked — Pending Approval.'
                             'Use Report ZR_SVC_WO_APPROVAL'
        RAISING blocked.

    WHEN OTHERS.
      MESSAGE i000(db) WITH 'Invalid approval status'
        RAISING blocked.
  ENDCASE.

ENDFUNCTION.
```

## 7. Anti-Patterns to Avoid

```abap
" ❌ WRONG: Displaying message directly inside FM (no RAISING)
" This shows a popup inside the FM — caller loses control
MESSAGE e398(00) WITH 'Error' lv_aufnr.

" ❌ WRONG: Checking sy-subrc after MESSAGE ... RAISING
" After RAISING, code below does not execute
MESSAGE i000(db) WITH 'Blocked' RAISING blocked.
IF sy-subrc = 0. ...  " Never reached

" ❌ WRONG: Using RAISE without MESSAGE
" sy-msg* fields are empty — caller cannot re-raise a meaningful message
RAISE insert_failed.

" ✅ CORRECT: Always pair MESSAGE ... RAISING together
MESSAGE e000(db) WITH 'Failed' iv_aufnr RAISING insert_failed.
```

## 8. Quick Reference Checklist

```
[ ] FM exceptions defined in interface (EXCEPTIONS section)
[ ] Each error path: MESSAGE <type><no>(<class>) WITH ... RAISING <exception>
[ ] OTHERS = <n> always included in CALL FUNCTION EXCEPTIONS
[ ] Caller checks: IF sy-subrc <> 0
[ ] Caller re-raises: MESSAGE ID sy-msgid TYPE sy-msgty NUMBER sy-msgno
                               WITH sy-msgv1 sy-msgv2 sy-msgv3 sy-msgv4
[ ] Use type 'E' for hard errors (insert_failed)
[ ] Use type 'I' for soft blocks (blocked / pending approval)
[ ] Never use MESSAGE without RAISING inside a FM called from BAdI/Exit
```
